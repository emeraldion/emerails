Detailed Description

Abstract base class for action controllers.

Project EmeRails - Codename Ocarina

Copyright (c) 2008, 2017 Claudio Procida http://www.emeraldion.it

A subclass of BaseController is in charge of handling a set of actions. It creates the network of model objects that will be rendered by the views, handles and validates postback data, defines and applies before and after filters.

BaseController::after_filter (   $filter,
  $params = NULL 
)
protected

Registers a function as a filter to be executed after the action is invoked.

EmeRails allows the developer to call one or more functions after the action has been invoked.

This allows the developer to perform manipulation of the response, to enable markup expansion, etc.

The controller keeps a list (FIFO) of filters that are invoked after the action method has returned. Everywhere in the filters chain, you can redirect the request or simply return silently to resign control to the next filter, or the response be flushed to the client. You can customize the way the filter is invoked by passing the optional parameter params.

Examples:

$this->after_filter(array('foo', 'bar'));

Executes the foo and bar filters (in this order) whatever action is requested.

$this->after_filter('foo', array('only' => 'index'));

Executes the foo filter only when the index action is requested.

$this->after_filter('foo', array('except' => array('index', 'list')));

Executes the foo filter unless the index or list actions are requested.

Parameters
filterThe name (or an array of names) of the method to be executed as filter. Currently, it must be a method of the controller.
paramsParameters that define when the filter has to be invoked.

protected

Registers a function as a filter to be executed before the action is invoked.

Action filtering

EmeRails allows the developer to call one or more functions before the action is actually invoked.

This may be useful to execute common code which is required before different action methods, to perform authentication checks, etc.

The controller keeps a list (FIFO) of filters that are invoked before the action method. Everywhere in the filters chain, you can redirect the request or simply return silently to resign control to the next filter, or the action itself. You can customize the way the filter is invoked by passing the optional parameter params.

Examples:

$this->before_filter(array('foo', 'bar'));

Executes the foo and bar filters (in this order) whatever action is requested.

$this->before_filter('foo', array('only' => 'index'));

Executes the foo filter only when the index action is requested.

$this->before_filter('foo', array('except' => array('index', 'list')));

Executes the foo filter unless the index or list actions are requested.

Parameters
filterThe name (or an array of names) of the method to be executed as filter. Currently, it must be a method of the controller.
paramsParameters that define when the filter has to be invoked.

BaseController::render (   $params)

Requests the rendering of the response.

Response rendering

This method is responsible of loading the view and layout relative to the current controller and action, parsing their content, and returning the result to be appended to the response.

The params array has the following semantics:

  • action: the name of the action for which render a view (defaults to current action).
  • layout: the layout used to render the view (optional, defaults to current action). It can be set to NULL to render the view with no layout (the same result can be obtained using the nolayout request parameter).
  • return: return the rendered view as a string, rather than flushing it to the output buffer.
  • partial: triggers the partial rendering mode (optional).
  • object: passes an argument to be used by the partial view (optional).
Parameters
paramsParameters defining how the rendering should be realized.

BaseController::render_page ( )

Renders the page as a response for the current request.

This method is responsible for building the response for the current request.

  • All before filters that are registered for the current action are executed.
  • If the current page is cacheable, and a cached version already exists in the caches, it is returned; otherwise it is rendered and eventually stored in the cache folder.
  • All after filters that are registered for the current action are executed.
  • Finally, the response is flushed to the client.
